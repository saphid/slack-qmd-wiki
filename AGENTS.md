# Slack-fed LLM Wiki Maintainer

This repository follows the persistent-wiki pattern from Andrej Karpathy's LLM knowledge-base gist: raw sources are immutable, the LLM-maintained wiki is the compiled knowledge layer, and this file is the schema/workflow contract.

## Repository layout

- `raw/slack/` — immutable-ish Slack source exports generated by `scripts/slack_sync.py`. Treat these as source of truth. Do not summarize by editing them.
- `inbox/` — generated ingest manifests. Each manifest lists new raw sources that need wiki integration.
- `wiki/` — LLM-owned markdown wiki.
  - `wiki/index.md` — content-oriented catalog. Update on every ingest.
  - `wiki/log.md` — append-only chronological operations log.
  - `wiki/pages/channels/` — channel pages.
  - `wiki/pages/people/` — person pages only when useful; avoid sensitive profiling.
  - `wiki/pages/topics/` — topic/project/concept pages.
  - `wiki/pages/decisions/` — decisions, rationale, links to Slack sources.
  - `wiki/pages/questions/` — open questions, risks, follow-ups.
  - `wiki/sources/slack/` — source summary pages derived from raw Slack batches.
- `scripts/` — ingestion/sync helpers. Scripts fetch and normalize; the LLM does semantic integration.

## Invariants

1. Raw Slack exports are evidence, not the wiki. Never rewrite them to make the wiki look cleaner.
2. The wiki is synthesized and maintained incrementally. Every ingest should update all touched pages, not just add a new source summary.
3. Every factual claim that came from Slack needs a citation to a raw source path and, when available, a Slack permalink.
4. Prefer durable knowledge over chat noise. Capture decisions, ownership, project state, unresolved questions, links, and named systems. Ignore routine acknowledgements and ephemeral chatter.
5. Respect privacy. Do not quote sensitive Slack content unless the quote itself is necessary evidence. Summarize minimally, cite precisely, and avoid creating people pages for gossip or performance judgments.
6. If new Slack contradicts an old wiki claim, update the claim and record the contradiction/history instead of silently overwriting context.

## Page conventions

Use Obsidian-style wiki links for internal links: `[[pages/topics/example-topic|Example topic]]`.

Recommended frontmatter:

```yaml
---
title: Page Title
type: topic|channel|person|decision|question|source-summary
status: active|stale|resolved|unknown
updated: YYYY-MM-DD
sources:
  - raw/slack/YYYY-MM-DD/channel-name.md
---
```

Citations should be compact and grep-friendly:

```md
Source: `raw/slack/2026-05-02/eng-platform.md` ts=1714591234.000100 permalink=https://...
```

## Slack ingest workflow

When asked to ingest Slack, or when processing an `inbox/slack-ingest-*.md` manifest:

1. Read `wiki/index.md` and recent `wiki/log.md` entries.
2. Read the manifest and then raw source files one at a time.
3. For each raw source:
   - Create/update one `wiki/sources/slack/YYYY-MM-DD-<channel>.md` source summary.
   - Update the channel page under `wiki/pages/channels/`.
   - Update/create relevant topic, decision, and question pages.
   - Add links between affected pages.
   - Update `wiki/index.md`.
4. Append a `wiki/log.md` entry with prefix `## [YYYY-MM-DD] ingest | Slack | ...` listing touched pages and unresolved follow-ups.
5. If the source is mostly noise, still log it briefly and say no durable updates were made.

## Query workflow

When answering questions against the wiki:

1. Read `wiki/index.md` first.
2. Search/read relevant pages and raw sources when claims need verification.
3. Answer with citations to wiki pages and raw Slack source paths/permalinks.
4. If the answer is useful as durable knowledge, offer to file it as a wiki page and update the index/log.

## Lint workflow

Periodic health check:

- Find orphan pages, stale pages, missing citations, duplicate topic pages, unresolved questions, and Slack source summaries that were not integrated.
- Update `wiki/index.md` and append a `lint` entry to `wiki/log.md`.
